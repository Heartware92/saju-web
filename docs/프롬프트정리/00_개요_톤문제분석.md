## 1. 한눈에 보는 구조

세 운세 모두 **AI에 보내는 메시지 = `시스템 프롬프트` + `유저 프롬프트`** 두 덩어리다.
- **시스템 프롬프트**: "너는 누구이고 어떤 톤으로 써라" (전역 지시)
- **유저 프롬프트**: 사주 데이터 블록 + 작성 규칙 + 공유 블록 + 섹션별 지침

```
                ┌─ 시스템 프롬프트 ─────────────┐   ┌─ 유저 프롬프트(본문) ──────────────────────┐
정통운세    →   │ DEFAULT_SYSTEM_PROMPT         │ + │ 사주데이터 + commonRules + METAPHOR_KB        │
(정통사주)      │ ("핵심만 간결·실용적으로")    │   │  + 은유제목규칙 + 핵심강조규칙 + 섹션지침      │
                └──────────────────────────────┘   └────────────────────────────────────────────┘
                                                     ↑ 2-pass: 1차 4섹션 → 2차 8섹션

신년운세    →   ┌──────────────────────────────┐   ┌────────────────────────────────────────────┐
연도별운세      │ NEWYEAR_SYSTEM_PROMPT        │ + │ 사주+세운데이터 + 작성규칙 + METAPHOR_KB      │
(동일)          │ ("간결 말고 분량 충족·풍부") │   │  + 은유제목규칙 + 섹션지침 + 분량강제          │
                └──────────────────────────────┘   └────────────────────────────────────────────┘
                                                     ↑ 2-pass: 1차 5섹션 → 2차 4섹션
```

### 운세별 핵심 사실

| 항목 | 정통운세 | 신년운세 | 연도별운세 |
|---|---|---|---|
| 프롬프트 생성 함수 | `generateJungtongsajuCorePrompt` + `generateJungtongsajuApplicationPrompt` | `generateNewyearReportPrompt` | **(신년과 동일 함수)** |
| 처리기(잡) | `jungtongsajuJob.server.ts` | `newyearJob.server.ts` | **(신년과 동일)** |
| 시스템 프롬프트 | `DEFAULT_SYSTEM_PROMPT` (aiClients.ts) | `NEWYEAR_SYSTEM_PROMPT` | 동일 |
| 구조 | 2-pass (4 + 8 섹션) | 2-pass (5 + 4 섹션) | 동일 |
| 섹션 수 | 12 | 9 | 9 |
| 크레딧 | 10달 | 10달 | 10달 |

> ★★ **신년운세 = 연도별운세는 프롬프트가 100% 같다.** `year` 값과 화면 라벨(`isYearFortune`)만 다르다.
> → **신년/연도별 톤을 고치면 두 서비스가 동시에 바뀐다.** 따로 손볼 필요 없음(따로 손볼 수도 없음).

### 정통운세가 다른 점 (2-pass의 의미)
- **1차(Core, 4섹션)**: general(총론)·daymaster(일주)·element(오행)·interaction(합충형파해) — 사주의 "기반 해석"
- **2차(Application, 8섹션)**: character·career·wealth·love·health·relation·luck·advice — 영역별 응용
- 1차 결과 전문을 2차 프롬프트에 넣어 "이미 쓴 표현·별칭 반복 금지"를 강제 → 그래서 2차 프롬프트에 **중복회피 규칙이 폭발적으로 많다.**

## 2. "국어사전 같다 / 접근하기 어렵다" — 원인 분석

피드백의 정체를 프롬프트에서 역추적하면, 톤을 그렇게 만드는 **구조적 장치 7가지**가 보인다. 각각 어디를 손대면 되는지까지 표시.

### 문제 ① 거대한 은유 사전(METAPHOR_KB)이 모든 섹션을 "문학적이지만 똑같이" 만든다  ★1순위
- 173줄짜리 고정 은유 사전(달·별·계절·12운성 등)을 **모든 본문 프롬프트에 통째로 삽입**(01 문서 B).
- 게다가 **은유 제목 규칙**(01-C)이 *모든 섹션 첫 줄*에 `[은유] ...` 시적 부제목을 강제.
- 결과: 매 섹션이 "잔잔한 호수 위의 첫눈" 류 시적 부제 + 별/달/계절 은유로 시작 → 화려하지만 **천편일률, 정작 내 얘기 같지 않음**. "AI가 시 쓰네" 인상.
- 역설: KB 안에 "같은 별칭 3회 이상 금지", "변형도 금지" 같은 **억제 규칙이 한가득**(688~710줄) — 은유를 잔뜩 넣으라 해놓고 반복은 막으니, 모델이 억지로 새 은유를 짜내며 더 어색해짐.
- 손댈 곳: METAPHOR_KB 삽입 여부/분량, METAPHOR_TITLE_RULE(은유 부제목 강제) 완화.

### 문제 ② 전문용어 "명시적 노출"을 강제한다 → 교재·사전 문체  ★2순위
- 섹션 지침마다 "**십성을 명시적으로 노출할 것**", "**일간 ◯◯ 기준 십성 대응을 본문에 명시**" 같은 지시가 반복(예: 02 문서 daymaster·element·career·wealth 섹션).
- "전문 용어 첫 등장 시 괄호로 쉬운 말 병기" 규칙과 합쳐지면 → **"편인(의심·분석 성향), 정재(안정적 재물)…" 식 괄호 범벅** → 딱 사전 표제어 풀이 느낌.
- 손댈 곳: "명시적 노출" 요구를 "필요할 때만, 쉬운 말 위주로"로 완화. 괄호 병기 빈도 제한.

### 문제 ③ 존댓말 "~합니다" 통일 + 단정 어투 강제 → 딱딱한 보고서체
- `SYSTEM_PROMPT`/commonRules: "~합니다체로 통일, ~해요 섞지 말 것"(prompts.ts 520), "흐린 표현 2회 이하, 단정적으로"(commonRules 6번).
- 결과: 친근한 구어체("~예요", "~거든요")가 원천 차단 → 강의록 톤.
- 단, 실시간 운세(today)는 이미 별도 시스템 프롬프트로 서술형을 쓰는 등 운세별 톤 분리 선례 있음 → 정통/신년도 톤 레버를 분리 조정 가능.
- 손댈 곳: 시스템 프롬프트의 문체 지시(존댓말 종류·단정성).

### 문제 ④ 글자수 최소치 강제 → 분량 채우기용 빈말·동어반복  ★신년/연도별 특유
- 신년 시스템 프롬프트가 "**최소 글자수 미달은 실패**"라고 못박음(01-E2). 섹션별 400~540자, 월별 흐름은 **2250~2900자**.
- 작성규칙 13번: "모든 단정 뒤에 '왜 그런지' + '그래서 어떻게'를 붙여 풍부하게" → **공식처럼 반복되는 3단 구성**이 12개월 내내 → 읽다 지침.
- 손댈 곳: 최소 글자수 하향 또는 "분량보다 밀도" 기조로 전환. monthly 분량 재검토.

### 문제 ⑤ "결론 먼저 + 근거 이어붙이기" 섹션 공식 → 모든 섹션이 같은 리듬
- commonRules 7번 / 신년 5번: "각 섹션 첫 문장에서 결론 먼저, 근거 이어붙이기".
- 정통 2차는 한술 더: 섹션마다 "1단락(도입)·2단락(…)·3단락(…)·4단락(…)" 단락 구성을 **번호로 지정**(02 문서 character~relation).
- 결과: 12섹션이 전부 똑같은 골격 → 글이 "양식" 같아짐.
- 손댈 곳: 단락 구조 지정을 느슨하게. 섹션마다 리듬 다양화 허용.

### 문제 ⑥ 규칙이 본문보다 많다 → 모델이 "규칙 지키기"에 매여 사람스러움 상실
- 정통 2차 프롬프트는 시작부터 1차 반복금지·별칭 0회 금지·키워드 등장횟수 제한(A옵션)·금지표현 패턴·자기검열 절차…가 **수십 줄**(02 문서 2644~2705).
- 데이터 무결성("0개 십성 언급 금지") 규칙도 곳곳에 중복.
- 이 규칙들은 *정확성*을 위해 필요했지만(신살 날조·없는 십성 언급 사고 방지), 부피가 커지며 **톤은 점점 방어적·경직**됨.
- 손댈 곳: 중복 규칙 통합·압축(정확성은 유지하되 분량 축소). 톤 지시가 규칙 더미에 묻히지 않게 상단 배치.

### 문제 ⑦ 구조화 포맷(라벨형) 섹션 → 폼 양식 느낌
- 정통 advice: `시간대:` `음식:` `평생 실천:` 불릿 등 **양식 채우기** 구조.
- 신년 lucky·monthly: "N월(등급·키워드) | 영역: ○○" 고정 헤더.
- 정보 전달엔 좋지만 "사람이 말해주는" 느낌과는 거리. (이건 의도적 UI 연동이라 톤 개선 우선순위는 낮음.)

## 3. 톤 레버 요약 (어디를 만지면 톤이 바뀌나)

| 레버 | 위치 | 영향 범위 |
|---|---|---|
| 문체(존댓말 종류·친근함·단정성) | 시스템 프롬프트 (DEFAULT / NEWYEAR), `SYSTEM_PROMPT` 톤 섹션 | 전역 |
| 은유 과잉 | `METAPHOR_KB`, `METAPHOR_TITLE_RULE` | 정통·신년·연도별 공통 |
| 전문용어 노출 강도 | 각 섹션 지침의 "명시적 노출" 문구 | 운세별 개별 |
| 분량/밀도 | `NEWYEAR_SYSTEM_PROMPT` 최소글자수, 섹션 글자수 범위 | 주로 신년/연도별 |
| 섹션 골격 획일성 | commonRules·섹션별 단락 번호 지정 | 운세별 개별 |

> ⚠️ METAPHOR_KB·은유제목·시스템프롬프트는 **다른 운세(궁합·자미두수·토정·실시간 등)도 공유**한다. 공통 블록을 건드리면 영향 범위가 넓으니, "정통·신년만" 바꾸려면 공통 블록을 복제·분기하거나 섹션 지침 레벨에서 조정하는 편이 안전.
