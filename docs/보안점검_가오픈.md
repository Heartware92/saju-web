## 1. 의존성 CVE (npm audit 결과) — 총 12건 (critical 0 / high 7 / moderate 4 / low 1)

`npm audit` 실행 결과. 운영(production) 의존성만 보면 6건(high 4, moderate 2).
**핵심: 실제 배포 앱에서 악용 가능한 것 우선. 나머지 다수는 빌드/개발 도구라 실위험 낮음.**

### 🔴 우선 수정 (런타임 영향)
- **next 16.1.6 → 16.2.9** — `Next.js: HTTP request smuggling in rewrites` (high)
  - 우리 런타임 프레임워크. rewrites 통한 요청 스머글링.
  - 수정: `npm i next@16.2.9` (16.x 마이너, package.json 범위 밖이라 `audit fix --force`가 권하는 버전)
  - ⚠️ **마이너 업그레이드라도 배포 전 반드시**: `npm run build` + `npx tsc --noEmit` + 핵심 흐름(가입·결제·운세생성) 수동 점검. 회귀 위험 때문에 **새벽 + 모니터링**.
- **axios 1.14.0 → 최신 1.x** — `Axios NO_PROXY Hostname Normalization Bypass → SSRF` (high)
  - 직접 의존(`^1.14.0`). SSRF는 사용자 입력 URL로 외부요청 시 위험. 우리 사용처 점검 필요(대부분 고정 엔드포인트라 실위험은 낮을 것).
  - 수정: `npm i axios@latest`

### 🟠 한 번에 정리 (대부분 빌드/전이 의존, 실위험 낮음)
`npm audit fix` 한 방으로 in-range 패치되는 것들:
- **ws 8.20.1** (high) — Memory exhaustion DoS. 전이 의존(supabase realtime 등). `npm audit fix`로 패치.
- **form-data** (high) — CRLF injection. 전이.
- **flatted** (high) — unbounded recursion DoS. 빌드/도구.
- **minimatch / picomatch / brace-expansion** (high/mod) — ReDoS·glob 매칭. 대부분 빌드 도구.
- **postcss <8.5.10** (mod) — CSS stringify XSS. **빌드 타임**(CSS 처리)이라 런타임 노출 거의 없음. next 업그레이드로 같이 해소.
- **follow-redirects / js-yaml** (mod) — 헤더 누출·DoS. 전이.
- **@babel/core** (low) — sourceMap 임의파일읽기. 개발 전용.

### 수정 순서 (새벽)
```bash
# 1) 안전한 in-range 패치 먼저
npm audit fix
# 2) axios 직접 업그레이드
npm i axios@latest
# 3) Next 마이너 업그레이드 (가장 위험 — 별도로, 테스트 후)
npm i next@16.2.9
# 4) 검증
npm run build && npx tsc --noEmit
# 5) 핵심 흐름 수동 테스트(가입→결제→운세) 후 배포
npm audit   # 잔여 확인
```
> ❌ `npm audit fix --force`는 **금지** — Next 메이저/범위밖까지 한 번에 올려 회귀 위험. 위처럼 개별·점진 적용.

## 2. Vercel Firewall (Pro 포함 — 추가 비용 0)

현재 firewall config 없음(기본 보호만). 메모리 방침: **보이지 않는 방어**(가시적 인터스티셜 X).

### 적용할 룰 (오탐 0 설계)
- **Rate limit (IP당, 고임계치)** — 정상 유저는 절대 안 걸리는 수준:
  - `/api/sms/send` : 예) 분당 10~20회/IP (정상=1~2회) — 솔라피 비용폭주·OTP 어뷰징 차단
  - `/api/auth/check-email`, 로그인, 결제 생성 : 보수적 임계치
- **악성 스캐너 경로 차단**(deny) — 정상 접근 자체가 없어 오탐 0:
  - `/wp-admin`, `/wp-login.php`, `/.env`, `/.git`, `/xmlrpc.php`, `/phpmyadmin` 등
- (선택) UA 없는/명백한 악성 봇 차단

### ❌ 적용하지 않을 것
- **OWASP CRS(Managed WAF) deny 모드** — 사주 텍스트·이름 특수문자·AI 프롬프트가 SQLi/XSS 패턴으로 **오탐 → 정상 결제/상담 차단** 위험. "큰일" 후보. (켜려면 반드시 **log 모드**로만)
- **Attack Challenge Mode** — 전 방문자에게 인증 화면 노출 → 전환율 타격 + 방침(인비저블) 위반.

### 적용 방법 (안전 절차)
1. **observe(log) 모드로 먼저 배포** → 차단 0, 기록만.
2. 하루 모니터링(오탐 없는지 Firewall 로그 확인).
3. 이상 없으면 **deny로 전환**.
- 적용: Vercel 대시보드 → 프로젝트(saju) → Firewall, 또는 Vercel API(`/v1/security/firewall/config`, `VERCEL_TOKEN` env 사용).

## 4-1. 응답 데이터 최소화 ("필요한 것만 던져라" — 친구 지적, 일부 타당)

친구 지적: "리스폰스에 user 정보를 다 던지면 안 되고 필요한 것만 줘야 한다."
→ **보안 취약점은 아님**(본인 데이터 + RLS 보호 검증 완료). 하지만 **데이터 최소화·페이로드 위생** 차원에선 맞는 말. 구분해서 처리:

### (A) 큰 user 객체 (id·email·user_metadata·identities) — 우리가 제어 못 함
- 출처: Supabase Auth `getSession()`/`getUser()` **표준 응답**(`useUserStore.ts:41`, `supabase.ts:28,131`).
- kakao avatar_url·provider_id·sub·phone·name 등은 **Supabase 인증 객체의 고정 스키마**라 우리가 필드를 못 줄임. **본인에게만** 가는 본인 데이터.
- 조치: 트리밍 불가. 단 **불필요하게 로깅/외부전송/콘솔출력 하지 말 것**(현재 그런 코드 없음). → 사실상 조치 없음.

### (B) 우리 쿼리의 `select('*')` — 최소화 가능 (실제 개선 포인트)
- `src/services/supabase.ts`에 클라이언트 `select('*')` **7곳**(user_credits·birth_profiles 등), `archiveService.ts`에도 birth_profiles 쿼리.
- `select('*')`는 모든 컬럼을 받아옴 → UI가 안 쓰는 컬럼까지 노출 + 페이로드 증가.
- 개선: **UI에 실제 필요한 컬럼만 명시** (예: `.select('moon_balance, updated_at')`).
- ⚠️ 위험도/우선순위 **낮음**: RLS로 남의 행은 못 읽고, 본인 데이터라 유출 아님. 회귀 위험만 주의(컬럼 빠뜨리면 UI 깨짐) → 화면별로 쓰는 필드 확인 후 점진 적용. **가오픈 필수 아님, 위생 개선.**
- 대상 라인(참고): `supabase.ts` 120·157·276·336·391·454·487, `archiveService.ts` 92·103.

## ✅ 확인된 안전 항목 (이번 세션 점검 완료 — 수정 불필요)

- **HTTPS/TLS**: `2000-saju.com` HTTP/2 + HSTS(max-age 2년) + HTTP→HTTPS 308 강제. 전송 암호화 완비.
- **RLS**: 비로그인(anon)으로 user_credits·birth_profiles·orders·credit_transactions·inquiries·otp_codes 등 **15개 테이블 전부 0행**(차단 확인). 남의 데이터 못 읽음.
- **service_role 키**: 클라이언트 비노출. 서버(API·서버컴포넌트 `share/[token]`)에서만 사용.
- **클라이언트 노출 키**: anon key·PortOne·Kakao JS키 — 전부 공개 설계 키(노출 무해).
- **비밀번호**: Supabase Auth bcrypt 해시. 평문 저장 안 함.
- **Vercel 플랜**: **Pro (active)** — 상업 운영 약관 OK + Firewall 사용 가능.
- **함수 리전**: 응답 헤더 `x-vercel-id: icn1`(Seoul) 확인 — Supabase(Seoul)와 동일 리전.
